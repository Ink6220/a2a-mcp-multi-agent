
# LLM Agent Base Module
This module defines a standard interface and implementation structure for creating Language Model (LLM) agents in the A2A (Agent-to-Agent) system. It includes:
- The base abstract agent class BaseAgent
- Core response and usage tracking models (ResponseFormat, Usage, etc.)
- Utilities for recording, parsing, and post-processing agent interactions

| Component                                | Type                | Purpose                                                                                           |
| ---------------------------------------- | ------------------- | ------------------------------------------------------------------------------------------------- |
| `BaseAgent`                              | Abstract Base Class | Defines a template for agent behavior, including streaming, invoking, tool use, and usage logging |
| `ResponseFormat`                         | Pydantic Model      | Describes standardized response schema from agents                                                |
| `Usage`, `ApiUsage`, `ExtraUsage`        | Pydantic Models     | Tracks token usage, tool calls, outputs, and timing                                               |
| `current_time_utc7_str`                  | Utility Function    | Returns the current ISO timestamp in UTC+7                                                        |
| `postprocess`, `_create_and_store_usage` | Methods             | Extract context details and log interaction usage                                                 |


## Core Components

### 1. `ResponseFormat` (Pydantic Model)

This model standardizes the structure of responses returned by agents. It ensures that all agents communicate in a predictable format, whether they are providing a final answer or delegating a task to another agent.

**Key Fields:**
*   `action`: (Literal: "answer", "call_next_agent") - The primary action the agent intends to take.
*   `status`: (Literal: "input_required", "completed", "failed") - The system-defined flow status of the task.
*   `message`: (str) - The message content for the caller (user or another agent).
*   `custom_status`: (Optional[str]) - For extended flow semantics (e.g., "hang_up").
*   `agent_name`: (Optional[str]) - Name of the agent to call if `action` is "call_next_agent".
*   `next_agent_instruction`: (Optional[str]) - Instruction for the next agent if `action` is "call_next_agent".
*   `artifacts`: (Optional[str]) - JSON-serializable structured data.

**Validation:**
*   Ensures `agent_name` and `next_agent_instruction` are provided if `action` is "call_next_agent".

### 2. Unified Provider System (via LiteLLM)
*   All agent implementations now use a single agent class (A2AOpenaiAgent) backed by a provider-agnostic architecture using LiteLLM. The provider is specified in the agent card and automatically handled at runtime.

| Provider    | Integration Module         | Environment Variables Required                                  |
| ----------- | -------------------------- | --------------------------------------------------------------- |
| OpenAI      | `openai_integration.py`    | `OPENAI_API_KEY`                                                |
| AWS Bedrock | `aws_integration.py`       | `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION_NAME` |
| Anthropic   | `anthropic_integration.py` | `ANTHROPIC_API_KEY`                                             |
| Google      | `google_integration.py`    | `GOOGLE_API_KEY`                                                |



### 3. Usage Tracking Models (Pydantic)

These models are designed for detailed logging and tracking of agent activity, particularly API calls and token consumption.

*   **`ExtraUsage`**: Captures additional token usage details like `reasoning_tokens` and `cache_tokens`.
*   **`ApiUsage`**: (Currently not directly used by `Usage` but defined) Intended for raw API call token counts.
*   **`Usage`**: The primary model for logging a complete interaction or step.
    *   `usage_id`: Unique identifier for the usage record.
    *   `context_id`: Identifier for the overall conversation or session.
    *   `task_id`: Identifier for the specific task being processed.
    *   `model_name`: Name of the LLM model used.
    *   `user_input`: The input query to the agent.
    *   `output`: The `ResponseFormat` object generated by the agent.
    *   `prompt_tokens`, `completion_tokens`: Token counts.
    *   `extra_usage`: Instance of `ExtraUsage`.
    *   `tool_calls`: List of `ToolCall` objects (from `a2a_mcp.common.types`).
    *   `tool_outputs`: List of `ToolOutput` objects (from `a2a_mcp.common.types`).
    *   `timestamp`: ISO format timestamp in UTC+7.

The `current_time_utc7_str()` utility function provides timestamps for these records.

### 4. `BaseAgent` (Abstract Base Class)

This is the cornerstone of the framework. Developers must inherit from `BaseAgent` to create concrete agent implementations.

**Initialization (`__init__`):**
*   `model_name` (str): The name of the LLM model this agent primarily uses.
*   `agent_card` (CustomAgentCard): An object (likely from `a2a_mcp.common.types`) describing the agent's capabilities, name, description, etc.
*   `card_discovery` (A2ACardDiscovery): An object responsible for discovering other agents in the network.

**Key Properties:**
*   `agent_name`: Name of the agent.
*   `description`: Agent's purpose.
*   `content_types`: Supported input content types.
*   `_usage_logs`: A dictionary to store `Usage` records.

**Abstract Methods (to be implemented by subclasses):**
*   `invoke(query, context_id, task_id, context)`: Handles a single, non-streaming query and returns a `ResponseFormat`.
*   `follow_up_invoke(query, context_id, task_id, context)`: Handles a follow-up interaction, typically after a delegated task is completed, to decide the next step.
*   `stream(query, context_id, task_id)`: Handles a streaming query, yielding responses incrementally.
*   `convert_tool_format(tools)`: Converts a generic tool definition into the format expected by the specific LLM or agent backend.
*   `_parse_to_response_format(self, data: Union[str, ResponseFormat]) -> ResponseFormat:`: Parses text output (potentially from an LLM) into a `ResponseFormat` object or returns the raw text if parsing fails.
*   `root_instruction(chat_history, tools, agent_info)`: Generates the main system prompt or instruction for the LLM.
*   `make_remote_agent_connection(target_agent_card, request)`: Establishes and manages a streaming connection to another agent.
*   `_extract_tool_calls_and_outputs(result)`: Extracts `ToolCall` and `ToolOutput` objects from an agent's processing result.
*   `_extract_tool_call(item)`: Helper to extract a single `ToolCall`.
*   `_extract_tool_output(item)`: Helper to extract a single `ToolOutput`.

**Concrete Methods:**
*   `record_usage(usage: Usage)`: Stores a `Usage` object in `_usage_logs`.
*   `get_usage_by_usage_id(usage_id: str)`: Retrieves a specific `Usage` record.
*   `postprocess(context: Dict[str, Task])`: Processes a context dictionary (containing tasks and their history) into a formatted string. This is likely used to construct a coherent history for the LLM. It handles various message parts (text, data, files) and tool interactions.
*   `_create_and_store_usage(...)`: A helper method to create a `Usage` object from various inputs (query, result, API usage details, tool calls/outputs) and then record it. It also prints usage details to the console using `colorama` for better visibility during development.

## How It Works (High-Level Flow)

1.  **Agent Specialization**: A developer creates a new agent class by inheriting from `BaseAgent`.
2.  **Implementation**: The developer implements all abstract methods, tailoring the agent's logic, LLM interaction, tool handling, and response parsing to its specific role.
3.  **Instantiation**: An agent instance is created with its model, agent card, and card discovery mechanism.
4.  **Invocation**:
    *   For a single request: The `invoke` method is called.
    *   For streaming: The `stream` method is called.
    *   For follow-ups: The `follow_up_invoke` method is called.
5.  **LLM Interaction**: Inside these methods, the agent would typically:
    *   Use `postprocess` to format the current `context` and `chat_history`.
    *   Use `root_instruction` to generate the system prompt.
    *   Call the underlying LLM.
    *   Use `_extract_tool_calls_and_outputs` if the LLM requests tool use.
    *   Use `_parse_to_response_format` to attempt to fit the LLM's text into the `ResponseFormat`.
6.  **Response Generation**: The agent constructs a `ResponseFormat` object.
7.  **Usage Logging**: Before returning, `_create_and_store_usage` is called to log the details of the interaction.
8.  **Inter-Agent Communication**: If `action` is "call_next_agent", the framework (or MCP) would use `agent_name` and `next_agent_instruction` to route the task. `make_remote_agent_connection` would be used if direct streaming to another agent is required.

## Provider Integration System

The A2A MCP framework now uses a unified provider integration system powered by LiteLLM, which allows seamless switching between different AI providers without changing the core agent logic.

### Supported Providers

The system supports the following providers through dedicated integration modules:

1. **OpenAI** (`openai_integration.py`)
   - Models: GPT-4, GPT-4-turbo, GPT-3.5-turbo, etc.
   - Required env var: `OPENAI_API_KEY`

2. **AWS Bedrock** (`aws_integration.py`)
   - Models: Amazon Nova, Claude on Bedrock, etc.
   - Required env vars: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION_NAME`

3. **Anthropic** (`anthropic_integration.py`)
   - Models: Claude-3-Haiku, Claude-3-Sonnet, Claude-3-Opus
   - Required env var: `ANTHROPIC_API_KEY`

4. **Google** (`google_integration.py`)
   - Models: Gemini-1.5-Flash, Gemini-1.5-Pro
   - Required env var: `GOOGLE_API_KEY`

### How It Works

1. **Agent Card Configuration**: Specify the provider in your agent card:
   ```json
   {
     "provider": {
       "organization": "aws"
     },
     "modelName": "amazon.nova-lite-v1:0"
   }
   ```

2. **Automatic Provider Selection**: The `A2AAgentSelector` automatically:
   - Detects the provider from the agent card
   - Validates the required environment variables
   - Loads the appropriate integration module
   - Converts the model name to LiteLLM format

3. **Unified Agent**: All providers use the same `A2AOpenaiAgent` class, which now works with any LiteLLM-compatible model string.

### Adding New Providers

To add support for a new provider:

1. Create a new integration module (e.g., `cohere_integration.py`)
2. Implement the validation function and integration class
3. Add the provider case to `A2AAgentSelector._get_model_integration()`

### Migration from Legacy System

The old system with separate `A2ANovaAgent` and `A2AOpenaiAgent` classes has been replaced with this unified approach. The `A2ANovaAgent` has been removed, and all functionality is now handled through the integration system.
